# Meccha Camouflage v1.4.0 Mesh-First Quality Plan

## Summary

The current mesh-first path proves that the live mesh, LOD0 profile, current
pose, BaseColor capture, and `ServerPaintBatch` replay can be connected, but
the planner is still wrong for production quality. It samples per triangle,
caps large triangles, uses a coarse brush/stamp model, exposes settings that no
longer match the runtime behavior, and still keeps old paint code around.

v1.4.0 should replace the current point-stamp planner with a UV-atlas coverage
planner. The production path should have one authoritative paint
implementation only: Mesh Profile V2 plus current skinned pose, UV atlas
coverage, source color transfer, and multiplayer replay through
`ServerPaintBatch`.

No fallback to dense hit-test, template paint, local-only paint, hidden
material fill, orbit capture, automatic quality downgrade, or local visual sync
should remain. If validation fails, paint fails clearly with Log and Trace
details.

## Current Findings

- The active mesh is already `paintman` LOD0:
  `1660` vertices, `8352` indices, `2784` triangles, `28` bones.
- The rough result is not an LOD problem. The current planner emits sparse
  triangle samples and paints them as large independent UV stamps.
- `planner_sample_step_texels=8` plus `brush_radius=0.01` means a roughly
  10 texel radius stamp on a 1024 texture, which is visually too coarse.
- `kMaxSamplesPerTriangle=64` makes large UV triangles under-sampled.
- `brush_spacing` is exposed in the UI but is not a real mesh-first coverage
  control.
- `server_brush_spacing` is written into brush settings, but the planner is
  sending independent UV strokes, so this setting does not mean "quality".
- The current side policy is nearest-UV color transfer. It can bleed because it
  lacks UV island/body-region guards.
- The old dense/template code is now legacy. It should be removed, not kept as
  fallback.

## Target Architecture

### 1. Runtime Resolve

Keep the current successful runtime resolve stages:

- resolve `RuntimePaintableComponent`
- select the live `paintman` `SkeletalMeshComponent`
- select the matching Mesh Profile V2 from `mesh-profiles/*.json`
- resolve current component-space pose
- resolve component-to-world transform
- resolve camera transform, viewport, and capture parameters
- resolve `ServerPaintBatch`

This stage must not generate paint samples and must not call
`HitTestAtScreenPosition`.

### 2. Mesh Profile V2

Extend the generated mesh profile consumed by runtime. The profile must contain
all data needed by the planner so runtime does not infer structure from ad hoc
string matching.

Required profile fields:

- mesh identity, asset path, export name, profile hash
- texture size used for planning, default `1024`
- LOD index, vertex count, index count, bone count
- vertices, indices, UV0, bone influences, reference bones
- triangle list with UV area, local normal, dominant bone, body region
- UV island id per triangle
- triangle adjacency by shared mesh edge and by UV edge
- body-region labels derived from dominant bones

Generated profiles remain build/release-prep artifacts. They are generated by
research tooling after each game update, copied to `mesh-profiles/*.json`, and
selected dynamically at paint time based on the live mesh identity. Full live
memory mesh extraction is deferred.

### 3. UV Atlas Coverage Planner

Replace `mesh_first_generate_plan_samples_from_runtime_cache` with a planner
that rasterizes mesh coverage in UV atlas space.

Rules:

- Use the runtime cached triangles for current world/local positions and UVs.
- Use the profile V2 triangle metadata for UV island, body region, adjacency,
  and expected identity.
- Convert the selected quality settings to texel-space planner values.
- For each triangle, compute its UV pixel bounding box.
- Step through UV texel cells using `coverage_step_texels`.
- Test each cell center against the triangle with barycentric coordinates.
- Emit one candidate per accepted UV cell with triangle id, UV, barycentric,
  local/world position, mesh region, source eligibility, UV island, and body
  region.
- Do not use a per-triangle sample cap. Only a global `max_strokes` validation
  exists, and exceeding it blocks replay with an actionable error.
- Do not downsample automatically. Automatic quality downgrade hides bugs.

This changes quality from "some samples per triangle" to "deterministic UV
coverage".

### 4. Region Model

Keep region names explicit and mesh-local:

- `mesh_front`
- `mesh_side`
- `mesh_back`

Default enabled regions:

- `mesh_front=true`
- `mesh_side=true`
- `mesh_back=true`

Classification:

- Destination region uses mesh-local normal and profile metadata.
- The front/back axis is selected from the profile bounds: use the thinner
  horizontal axis as character depth. For `paintman`, this is `local_y`; `local_x`
  is body left/right and must not be used as front/back.
- Color source eligibility uses current screen projection plus a lightweight
  skinned-mesh depth filter, not a world-normal prefilter.
- Source selection must be independent from destination. A back destination can
  receive color from the same shared camera-facing source set.

Back behavior:

- `mesh_back` is not hidden fill.
- `mesh_back` uses the same captured source color model as `mesh_front`.
- No white metallic prefill or material pass is part of v1.4.0.

Side behavior:

- `mesh_side` must be conservative.
- Side samples may only use sources from compatible UV island/body region.
- Side samples must satisfy a strict source-distance guard.
- Unsafe side samples block replay when side is enabled. They are not skipped.

### 5. Source Color Capture

Source samples are camera-facing, projected mesh candidates.

Rules:

- Do not call `HitTestAtScreenPosition`.
- Project source candidates into the current capture view.
- Drop source candidates outside the capture view before color transfer.
- Build a lightweight screen-space depth filter from the projected skinned mesh
  and keep only frontmost source candidates. This avoids mixing self-occluded
  mesh faces into the source color set without going back to runtime hit tests.
- If no valid source samples remain, fail before replay.
- Capture BaseColor through the existing SceneCapture/bulk readback path.
- Store sampled colors as source records keyed by UV, triangle, body region,
  UV island, mesh region, and world position.

Capture failure is a hard error. It must not switch to old hit-test sampling.

### 6. Color Transfer

Replace raw nearest-UV lookup with guarded source selection.

For each destination candidate:

- prefer source candidates with the same body region
- prefer source candidates with the same mesh-local region, then fall back to
  the shared visible source set
- prefer same UV island when available
- require side candidates to stay under `side_source_max_uv`
- require front/back candidates to stay under `front_back_source_max_uv`
- use a UV-grid nearest compatible source lookup rather than brute-force
  destination x source scanning
- if no compatible source exists, mark the candidate unsafe

Replay is blocked when any unsafe candidate belongs to an enabled region.
Trace must report unsafe counts by region and reason.

### 7. Stroke Generation

Use one production replay path only: `ServerPaintBatch`.

Stroke rules:

- Target channel is `Albedo`.
- `FPaintStroke.Uv` is used.
- `bHasWorldPosition=false`.
- `bHasLocalPosition=false`.
- `bHasSkeletalTriangleAnchor=false` unless a later validated server need
  proves otherwise.
- `PaintAtUVWithBrush` is not used in production replay or local mirror sync.
- `ServerPaintBatch` is the only authoritative replay operation.
- `BrushSettings.Radius` comes from `stroke_radius_texels / texture_size`.
- `BrushSettings.Spacing` is fixed internally and not exposed as quality.
- `Hardness=1.0`, `Opacity=1.0`, normal blend mode.

The planner controls density through UV coverage cells, not through
`BrushSettings.Spacing`.

### 8. Settings And UI

Replace the current Paint Settings fields with settings that match mesh-first
behavior.

Remove or migrate:

- remove `Brush spacing`
- remove `Server spacing`
- stop presenting spacing as a quality control

Add:

- `Quality preset`: `Balanced`, `High`, `Ultra`
- `Stroke size`: texels, default `4.0`; replay converts this to normalized UV
  brush radius using the selected profile texture size.
- `Coverage step`: texels, default `6.0`
- `Side guard`: UV distance, default `0.08`
- `Front/back guard`: UV distance, default `0.45`
- `Max strokes`: default `25000`
- region toggles: `Mesh front`, `Mesh side`, `Mesh back`
- server safety: `Batch limit`, `Batch delay ms`

Preset defaults:

- `Balanced`: stroke `5.0`, step `8.0`, max strokes `15000`
- `High`: stroke `4.0`, step `6.0`, max strokes `25000`
- `Ultra`: stroke `3.5`, step `4.0`, max strokes `50000`

Default preset for v1.4.0 is `High`.

UI behavior:

- Save/edit/cancel behavior remains unchanged.
- Hotkey execution remains disabled while Paint Settings are being edited.
- Paint Settings should show the saved effective values only.
- Info should show last plan summary: source count, stroke count, batch count,
  estimated replay time, unsafe count.
- Trace contains raw planner details. Log contains user-facing failures.

### 9. Batch And Performance

Server safety stays explicit.

Rules:

- Use only configured `server_batch_limit` and `server_batch_delay_ms`.
- Do not silently reduce delay.
- Do not silently reduce quality to fit time.
- Before replay, estimate:
  - stroke count
  - batch count
  - estimated replay time
- If `stroke_count > max_strokes`, block replay and ask the user to lower
  quality or region coverage.

Default server settings remain:

- `server_batch_limit=50`
- `server_batch_delay_ms=300`

This may be slow in `High` or `Ultra`, but it is predictable and avoids hiding
server-load problems.

### 10. Legacy Removal

Remove production code paths that are no longer part of v1.4.0:

- dense hit-test sampler
- `template_brush_paint` route
- template phase sample pool
- `PaintAtScreenPosition` based paint
- hidden material fill
- local `PaintAtUVWithBrush` replay
- automatic fallback modes
- stale UI fields for old brush spacing semantics

`HitTestAtScreenPosition` may remain only if required by unrelated reflection
metadata or research-only code, but production mesh-first paint must not call it
or advertise it as part of the runtime path.

## Implementation Phases

### Phase 1: Settings Schema

- Add `MeshPaintQuality` / `PaintTuning` fields for the new settings.
- Bump config layout version.
- Migrate old configs by resetting paint settings to the new defaults.
- Remove old spacing fields from UI and payload.
- Keep server batch settings.

### Phase 2: Profile V2 Generation

- Extend `scripts/research/prepare-mesh-profile.ps1` / research export flow to
  generate Mesh Profile V2 with UV islands, triangle local normal, dominant
  bone, and body region. Triangle adjacency can remain profile metadata for
  future planner improvements.
- Update the runtime profile loader to require profile V2.
- Fail on profile V1 or missing metadata.

### Phase 3: UV Atlas Planner

- Implement UV triangle rasterization over `coverage_step_texels`.
- Build candidate records with region/source/body/island metadata.
- Remove per-triangle sample caps.
- Add global max stroke validation.
- Add planner metrics to Trace.

### Phase 4: Guarded Color Transfer

- Build source indexes by body region and UV island.
- Implement nearest weighted compatible source lookup.
- Track unsafe reasons:
  - no compatible source
  - side source distance exceeded
  - front/back source distance exceeded
  - UV island mismatch
  - body region mismatch
- Block replay if enabled regions contain unsafe candidates.

### Phase 5: Server Replay Cleanup

- Send exactly one Albedo `ServerPaintBatch` stream.
- Drain the stream through the game thread message/timer loop, one batch per tick, so batch pacing never sleeps on the game thread.
- Do not run local visual sync after server replay.
- Report batch counts and elapsed time.
- Keep channel checksum diagnostics for Albedo.

### Phase 6: Legacy Code Deletion

- Delete obsolete dense/template implementation after mesh-first build passes.
- Remove stale CLI/native mode references.
- Remove old docs that describe the legacy production path.

### Phase 7: Runtime Validation

- Run live tests at front, diagonal, side-biased, and tilted angles.
- Validate front/back same-source behavior.
- Validate side guard does not visibly cross the side boundary.
- Validate roughness caused by huge brush is gone.
